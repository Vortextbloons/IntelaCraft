# Development Guide

## Prerequisites

- Node.js v20+ (ES modules)
- npm (workspaces)
- Minecraft Bedrock Dedicated Server (for testing)

## Setup

1. Clone repository
2. Run setup:
   ```powershell
   npm run setup
   ```
   This installs dependencies, builds all packages, and creates `.env`.
3. Start controller:
   ```powershell
   npm run dev
   ```
   Controller starts at http://127.0.0.1:8787
4. Open webview at http://127.0.0.1:8787

## Project Structure

Monorepo with npm workspaces:
- `apps/` - Bedrock addon, webview
- `packages/` - Shared libraries (shared-protocol, pi-extension, etc.)
- `services/` - Controller, API services

Each workspace has its own `package.json` and `tsconfig.json`.

## Coding Conventions

- TypeScript strict mode
- ES Modules (`"type": "module"` in package.json)
- No external HTTP frameworks (raw `node:http`)
- React 19 with hooks (no state management library)
- esbuild for bedrock addon bundling

## Adding New Tools

1. Add tool definition to `shared-protocol/constants.ts` (ToolName)
2. Add argument type to `shared-protocol/types.ts`
3. Add validation to `shared-protocol/validate.ts`
4. Add factory to `shared-protocol/factory.ts`
5. Implement in bedrock-addon (`inspect/` or `mutate.ts`)
6. Add to pi-extension PLANNER_TOOL_CATALOG if inspect tool

## Adding New API Endpoints

1. Add route handler in `services/controller/src/app.ts`
2. Add to API reference docs

## Adding New Packages

1. Create directory in `packages/`
2. Add `package.json` with `@intelacraft` scope
3. Add `tsconfig.json`
4. Add to root `package.json` workspaces

## Code Style

- No comments unless asked
- Follow existing patterns
- Prefer editing existing files
- Use TypeScript types everywhere

## Scripts

- `dev.ps1` - PowerShell launcher for all commands
- `scripts/*.mjs` - Shared CLI tools
  - `setup.mjs` - Install deps, build, create `.env`
  - `dev.mjs` - Start controller + Vite dev server
  - `deploy.mjs` - Build and deploy Bedrock packs
  - `health.mjs` - Check controller + BDS connection
  - `inspect.mjs` - CLI tool to run inspect queries against a connected BDS
  - `configure-bds.mjs` - Write BDS config files
  - `load-smoke.mjs` - Concurrent poll/enqueue smoke test
  - `combine-docs.mjs` - Combine all docs into a single markdown file

## Environment Setup

### .env Locations

The controller loads `.env` from the project root. A `.env.example` is generated by `npm run setup`:

```powershell
# Project root (primary config)
.env              # Controller environment variables

# BDS config (generated by deploy)
apps/bedrock-addon/config/variables.json
apps/bedrock-addon/config/secrets.json
apps/bedrock-addon/config/permissions.json
```

### Required Variables

At minimum, set these in `.env`:

```
INTELACRAFT_BDS_TOKEN=dev-change-me
```

Provider settings can be configured later via the webview Settings → Providers page, or set upfront:

```
INTELACRAFT_PROVIDER_BASE_URL=https://api.openai.com/v1
INTELACRAFT_PROVIDER_API_KEY=sk-...
INTELACRAFT_PROVIDER_MODEL=gpt-4o
```

See `docs/reference/configuration.md` for the full variable reference.

### VS Code Recommended Extensions

- **Error Lens** - Inline error display

### TypeScript Configuration

Each workspace has its own `tsconfig.json` extending the root config. The root `tsconfig.json` defines shared compiler options:

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "Node16",
    "moduleResolution": "Node16",
    "strict": true,
    "esModuleInterop": true,
    "skipLibCheck": true
  }
}
```

To resolve `@minecraft/*` types, ensure your IDE can find the BDS type definitions. The `apps/bedrock-addon/tsconfig.json` includes paths to the BDS types directory.

## Monorepo Structure

### How Workspaces Link

npm workspaces use symlinks in the root `node_modules/` directory. When you run `npm install`, dependencies of each workspace are hoisted to the root. Cross-workspace references (e.g., `@intelacraft/shared-protocol`) are symlinked directly.

### Workspace Protocol

Workspaces reference each other using npm workspace protocol:

```json
{
  "dependencies": {
    "@intelacraft/shared-protocol": "workspace:*"
  }
}
```

The `workspace:*` specifier means "use the local version from the monorepo."

### Adding a New Package

1. Create directory in `packages/`
2. Add `package.json` with `@intelacraft` scope:

```json
{
  "name": "@intelacraft/my-package",
  "version": "0.0.0",
  "type": "module",
  "main": "dist/index.js",
  "types": "dist/index.d.ts"
}
```

3. Add `tsconfig.json` extending root config
4. Add to root `package.json` workspaces array
5. Run `npm install` to link

## Webview Development

### Vite Dev Server

The webview uses Vite for fast development. During development, the controller proxies `/v1/*` requests to the backend:

```typescript
// apps/webview/vite.config.ts
export default defineConfig({
  server: {
    proxy: {
      '/v1': 'http://127.0.0.1:8787',
    },
  },
});
```

### Hot Reload

Vite provides instant hot module replacement (HMR). Changes to React components update the browser without a full reload.

### Building for Production

The webview builds to `dist/` and is served by the controller at `/`. The build uses:

```powershell
cd apps/webview
npm run build
```

## Debugging Techniques

### Console Output

Use `console.log()`, `console.error()`, and `console.warn()` in any package. Output appears in the terminal running `npm run dev` for the controller, or in the BDS console for the bedrock addon.

### Controller Logs

The controller logs to stdout. Key events logged:

- Incoming requests (method, path)
- Action lifecycle (created, approved, executed, completed)
- Provider calls and responses
- Error stack traces

### BDS Logs

Check the BDS server console for `[IntelaCraft]` prefixed messages. These include:

- Session handshake results
- Action execution results
- Error messages from the addon
- Heartbeat status

### Audit Log

All mutations are recorded in the JSONL audit log at `INTELACRAFT_AUDIT_PATH`. Each line is a JSON object with timestamp, action, risk, and result:

```json
{"ts":"2026-07-10T12:00:00Z","action":"world.fill_blocks","risk":"normal","result":"success","blocks":128}
```

### Pi Session Debugging

Check the Pi session directory (`INTELACRAFT_PI_STORAGE_PATH`) for:

- `session.json` - Session metadata and status
- `messages.jsonl` - Full conversation history with the AI
- `plan.json` - The generated plan (if planning completed)
